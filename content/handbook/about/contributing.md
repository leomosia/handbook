The handbook is improved through pull requests.

Contributing to the Leo Mosia Handbook is not just about editing pages. It is part of how we operate.

Every contribution should make the system clearer, stronger, easier to use, and easier to maintain. The goal is not to add more content—it is to improve the operating manual that helps Leo Mosia think, build, decide, and operate with greater clarity.

Contributors are expected to work like Leo Mosia operators: make focused changes, explain the reason behind them, respect the source of truth, and leave the handbook better than they found it.

Outside contributors are always welcome. If you notice something that can be corrected, clarified, improved, or strengthened, feel free to open a pull request. Good external contributions are reviewed with the same care and seriousness as internal operator contributions.

## Operator Mindset

A Leo Mosia operator contributes with care, clarity, and ownership.

This means:

- improving the system, not just adding content
- reducing confusion rather than creating it
- documenting knowledge where it belongs
- respecting existing structure before creating new structure
- making focused, meaningful changes that are easy to review
- thinking about the future operators who will rely on this handbook
- treating documentation as part of the work, not separate from it

If something is unclear, outdated, duplicated, or missing, the operator response is to improve the handbook.

## Good Contributions

Good contributions usually do one or more of the following:

- Clarify an existing process.
- Add missing context.
- Move content to its correct source of truth.
- Remove outdated or duplicated material.
- Link related handbook pages together.
- Improve structure, naming, or navigation.
- Fix broken links, formatting issues, or inaccurate references.
- Turn repeated explanations into reusable handbook content.
- Make a page easier for a new operator to understand and use.

## Pull Request Expectations

Pull requests should be focused and easy to review.

A good pull request should:

- solve one clear problem
- explain why the change is needed
- update the appropriate page or section
- avoid duplicating another source of truth
- use clear, concise language
- preserve the voice and structure of the handbook
- request review from the appropriate owners

Large, unclear, or unfocused changes are difficult to review and maintain. Whenever possible, break them into smaller pull requests.

## Review

The reviewer is responsible for ensuring the change is clear, useful, and belongs in the right place.

Before approving a change, ask:

- Does this improve the handbook?
- Does it belong here?
- Is this the correct source of truth?
- Does it duplicate another page?
- Will future operators understand it?
- Does it reflect the Leo Mosia way of operating?

Review is more than approval. It protects the quality, structure, and long-term usefulness of the handbook.

## Standard

The handbook should become more useful with every contribution.

Do not contribute for the sake of activity. Contribute to create clarity, preserve knowledge, improve operations, and strengthen the system.
